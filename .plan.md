# Implementation Plan: Convert Opponent Hidden Hands from Fan Layout to Hand Rail

## Context Reviewed
- `ARCHITECTURE.md`: not present
- `AGENTS.md`: not present
- `CLAUDE.md`: not present
- `README.md`: present, but generic Vite/React boilerplate and not architecture-specific

## Files That Need to Change
- `src/ui/components/PlayerPanel.tsx`
- `src/ui/arena-theme.css`
- `qa/new-game-hand-rail.md`
- `qa/hidden-hand-commander-fanout.md`

## Goal
Render both the local player hand and opponent hidden hands using the same horizontal hand-rail layout. Opponent hands must remain hidden, so their hand cards should render as card backs in the rail instead of a fan.

## Current Implementation Summary
- The local seat uses a rail built from `buildHandRailCards(...)`.
- Hidden opponent seats do not render their `Zone.HAND` cards in the rail.
- Hidden opponent hands instead render through `HiddenHandFan` in `PlayerPanel.tsx`.
- The commander for hidden seats is pulled out as a separate visible rail card and visually attached to the fan.
- `arena-theme.css` contains dedicated hidden-fan layout, spacing, and responsive overrides.
- `GameBoard.tsx` already treats hidden-hand movement specially for battlefield-entry effects, and hidden hand art is already excluded from prefetching there.

## Desired End State
- One rail implementation for all seats.
- Visible local hand: unchanged behavior, still shows real cards in the rail.
- Hidden opponent hand: render one hidden placeholder per actual hand card in the same rail.
- Visible commander/exile/graveyard cards still appear in the rail and keep their zone-specific styling.
- No fan-specific renderer, no fan-specific CSS, and no fan-specific QA expectations.
- Hidden-hand placeholders do not reveal card information, do not trigger art loading, and do not gain action or drag affordances.

## Step 1: Replace the visible-card-only rail model with a unified rail item model
**File:** `src/ui/components/PlayerPanel.tsx`

**What to change**
- Replace `HandRailCard` with a discriminated union that can represent both visible rail cards and hidden hand placeholders.
- Suggested shape:
  - `{ kind: 'card'; card: CardInstance; railIndex: number }`
  - `{ kind: 'hidden-hand'; objectId: string; railIndex: number }`
- Replace `buildHandRailCards(...)` with a new builder that returns unified rail items instead of only `CardInstance[]`.
- Preserve ordering as:
  1. command cards
  2. hand cards or hidden-hand placeholders
  3. playable exile cards
  4. playable graveyard cards

**Why**
- The current model cannot represent hidden hand cards in the rail without filtering them out entirely.
- A unified item model allows one renderer and one layout path for all seats.

**Important details**
- For hidden seats, emit one placeholder per actual hand card.
- Use the underlying hand card `objectId` as the placeholder key; it is already present in client state and is the most stable identifier available.
- The builder should decide visibility based on `seat.handHidden`, not by post-filtering a visible-only list later.
- Keep playable exile/graveyard filtering exactly as it works now via `collectPlayableCardIds(...)`.

## Step 2: Remove the fan-only rendering path and move all seats to one rail
**File:** `src/ui/components/PlayerPanel.tsx`

**What to change**
- Delete `HiddenHandFan`.
- Remove fan-only derived values:
  - `hiddenFeaturedRailCard`
  - `visibleRailCards`
  - `showHiddenHandFan`
  - `data-has-fan`
- Change `hasRailContent` to be based only on whether the unified rail item list is non-empty.
- Render a single `.arena-seat__hand-cards` list for every seat.

**Why**
- The fan is the behavior being removed.
- Keeping parallel render paths would preserve unnecessary complexity and make regressions more likely.

**Important details**
- Do not special-case the commander anymore; it should simply be another visible rail card ordered with the other `Zone.COMMAND` cards.
- Preserve the existing empty-state behavior when there are no command, hand, playable exile, or playable graveyard items.
- Keep the existing `.arena-seat__hand-area` `HAND` anchor registration on the container.

## Step 3: Render visible cards and hidden placeholders through separate branches
**File:** `src/ui/components/PlayerPanel.tsx`

**What to change**
- Replace `renderRailCard(...)` with either:
  - one `renderRailItem(...)` switch, or
  - separate helpers for visible cards and hidden placeholders
- For visible items:
  - keep using `CardView`
  - keep preview, hover scaling, drag behavior, zone anchors, and mount refs
- For hidden placeholders:
  - render a non-interactive hand-sized card-back shell
  - do not use `CardView`
  - do not attach preview, click, drag, focus, or zone-anchor behavior
  - do not call `registerCardElement(...)`

**Why**
- Hidden opponent cards must remain hidden and must not trigger preview, art loading, actions, or drag state.

**Important details**
- Hidden placeholders should still participate in normal rail overlap and z-index stacking.
- Do not attach `onMouseEnter` to hidden placeholders; only visible cards should affect `hoveredHandIndex`.
- Keep `onMouseLeave={() => setHoveredHandIndex(null)}` at the hand-area level.
- Reuse the existing `.arena-card` / `.arena-card__frame` / `.arena-card-back` structure for hidden placeholders so they inherit current sizing and border-radius conventions cleanly.
- Preserve a hand-count affordance for hidden seats after removing the fan count badge, for example via a `title` on the hand area or rail when `seat.handHidden` is true.

## Step 4: Keep zone anchors and animation behavior stable
**File:** `src/ui/components/PlayerPanel.tsx`

**What to change**
- Build `anchorCardIds` from visible rail cards only.
- Keep the `HAND` zone anchor on `.arena-seat__hand-area`.
- Keep `COMMAND`, `EXILE`, and `GRAVEYARD` anchors on the first visible rail card from those zones.

**Why**
- `GameBoard.tsx` uses these anchors for battlefield-entry animations.
- Hidden placeholders must not become anchors for visible-zone animations.

**Important details**
- Preserve the current keying scheme passed to `registerZoneAnchor(...)`.
- For `EXILE` and `GRAVEYARD`, continue the current behavior where the first visible playable rail card can override the pile anchor for animation origin.
- No `GameBoard.tsx` change should be required initially, because hidden-hand battlefield entry already originates from the `HAND` container anchor.
- During verification, explicitly check battlefield-entry animations from hidden `HAND`, visible `COMMAND`, visible `EXILE`, and visible `GRAVEYARD` sources. If a regression appears, handle it as a follow-up refinement rather than expanding the initial scope.

## Step 5: Normalize z-index and hover behavior for the unified rail
**File:** `src/ui/components/PlayerPanel.tsx`

**What to change**
- Keep `railIndex` on every rail item and use it consistently for stacking order.
- Preserve the current hover-lift behavior for visible cards only.
- Remove the hidden-seat z-index assumptions that only existed to support the fan-plus-featured-card composition, if they are no longer necessary after unification.

**Why**
- The current stacking logic is partly compensating for the split fan layout.
- A single rail needs predictable overlap regardless of whether an item is visible or hidden.

**Important details**
- The hovered visible card should still rise above adjacent rail items.
- Hidden placeholders should remain static and should not appear actionable on hover.
- Mixed rails such as `COMMAND + hidden hand placeholders + EXILE` must stack in a stable left-to-right order.

- Ensure the hidden placeholder:

  - visible `.arena-card`

**Why**

**Important details**

- Hidden placeholders should not rotate or arc.
- Hidden placeholders should not show hover transforms, focus rings, or cursor styles that imply interactivity.

## Edge Cases To Handle
- Hidden seat with `hand.length === 0` but a commander in `Zone.COMMAND`:
  - the rail should still render the commander
  - no empty-state pill should appear
- Hidden seat with no hand, no commander, and no playable exile/graveyard cards:
  - show the existing `No cards ready` empty state
- Hidden seat with a large hand:
  - rail becomes horizontally scrollable
  - no clipping against zone piles or viewport edges
- Multiple command cards:
  - preserve current `command` array order
  - first visible command card remains the `COMMAND` anchor
- Playable exile or graveyard cards on hidden seats:
  - remain visible in the rail after the hidden hand placeholders
  - keep their distinct zone border treatment
- Visible local seat:
  - remains unchanged in behavior and interactivity
  - still supports preview, hover lift, and dragging
- Hover behavior:
  - visible cards still enlarge/lift correctly
  - hidden placeholders do not reveal card information or feel actionable
- Touch devices:
  - hidden placeholders must look correct without hover
  - tap interactions should remain limited to visible cards
- Hidden-hand privacy/performance:
  - hidden hand placeholders must not use `CardView`
  - hidden hand placeholders must not trigger card-art fetches or previews

## Potential Issues / Risks
- **Rail width growth:** hidden opponents now render one placeholder per card, so long hands can produce a much wider rail than the fan did.
- **Trailing spacing bugs:** if the last-item margin fix only handles visible cards, hidden rails may show extra blank width.
- **Z-index regressions:** mixing visible cards and hidden placeholders in one rail can create stacking mistakes if `railIndex` is not applied consistently.
- **Commander styling regression:** once the commander is no longer treated as a featured fan card, it must remain visually distinct via existing `data-source-zone="COMMAND"` styling.
- **Interaction leakage:** if hidden placeholders reuse visible wrapper behavior too aggressively, they may accidentally respond to hover, preview, or drag.
- **Animation origin mismatch:** keeping the `HAND` anchor on the container should be acceptable, but hand-to-battlefield animation origin should still be checked after the layout change.
