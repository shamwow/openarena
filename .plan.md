## Target Behavior
- Every seat renders one hand rail container.
- Hidden opponent `HAND` cards remain concealed, but each occupies a rail slot as a card back.
- `COMMAND` cards stay visible and appear in the same rail as hidden hand slots.
- Playable `EXILE` and `GRAVEYARD` cards still render visibly in the same rail.
- Hidden placeholders are non-interactive: no preview, click, drag, focus affordance, or per-card zone-anchor behavior.
- No fan markup, featured-card wrapper, fan count badge, or fan-specific spacing remains.
- Hidden hand size remains visually legible from the number of card backs and textually exposed through a simple affordance such as `title` or accessible label.

## Step-by-Step Plan

### 1. Replace the fan-only rail model with a unified ordered rail item model
**File:** [`src/ui/components/PlayerPanel.tsx`](/Users/ironsha/Desktop/projects/openarena/.worktrees/orchestrate-1774489757000/src/ui/components/PlayerPanel.tsx)

**What to change**
- Remove the `HiddenHandFan` component.
- Replace the current `HandRailCard` shape with a discriminated union for ordered rail items.
- Use a shape equivalent to:
  - `{ kind: 'card'; card: CardInstance; railIndex: number }`
  - `{ kind: 'hidden-hand'; key: string; railIndex: number }`
- Rename the current builder so its name reflects that it returns rail items rather than only visible cards.

**Why**
- The current model cannot represent hidden hand slots within the rail.
- A discriminated union keeps hidden placeholders out of the interactive `CardView` path.

### 2. Build one rail in explicit zone order for every seat
**File:** [`src/ui/components/PlayerPanel.tsx`](/Users/ironsha/Desktop/projects/openarena/.worktrees/orchestrate-1774489757000/src/ui/components/PlayerPanel.tsx)

**What to change**
- Build rail items in this order:
  1. `COMMAND`
  2. `HAND`
  3. playable `EXILE`
  4. playable `GRAVEYARD`
- For the local seat, `HAND` contributes visible card items.
- For hidden seats, `HAND` contributes one hidden placeholder item per card.
- Assign `railIndex` only after the ordered rail list is complete.

**Why**
- This preserves the current product ordering while making all seats structurally consistent.
- Final index assignment keeps lift and stacking aligned to the actual visual order.

## Edge Cases to Account For
- Hidden seat with `HAND` cards but no `COMMAND`: concealed rail still renders without extra spacing.
- Hidden seat with `COMMAND` but empty `HAND`: commander still appears in the rail and no fan gap remains.
- Hidden seat with playable `EXILE` or `GRAVEYARD`: visible cards appear after hidden placeholders in one ordered rail.
- Large hidden hands: rail scrolls horizontally instead of shrinking to unusable widths or overflowing the seat.
- Visible cards at rail edges: hover lift remains visible and unclipped.
- Hidden placeholders at rail edges: remain static and inert.
- Tablet and mobile layouts: opponent rails do not push surrounding seat UI off-screen.
- Zone anchors: only real cards register per-card anchors for `COMMAND`, `EXILE`, and `GRAVEYARD`.

## Potential Issues
- Hidden placeholders may size incorrectly if they do not inherit the same dimensions as `.arena-card[data-variant="hand"]`.
- Assigning `railIndex` too early may break stacking and hover presentation.
- Shared hover handlers may accidentally animate neighboring cards when hovering placeholders.
- Dead fan CSS may leave phantom offset, overlap, or padding behavior.
- Removing the fan badge without a replacement label/title may silently regress count affordance.

## Testing Strategy

### Code-level verification

### Product-level validation requirements
- Validation must not stop at build and lint.
- The reviewer must be able to load the running product into the target state and verify the change on screen.
- Because this is an interactive web UI change, validation must use Playwright.
- Validation must include a Playwright-captured video showing the interaction behavior, not just screenshots or DOM assertions.

### Product setup steps for reviewers
1. Start the app locally using the repo’s normal dev or preview command.
2. Open the app in a browser with Playwright.
3. Load the product into a game state that exposes all hand-rail cases needed for review:
   - local player with visible hand cards
   - at least one hidden opponent with concealed hand cards
   - at least one visible commander in an opponent rail
   - ideally at least one playable `EXILE` or `GRAVEYARD` card in a rail
4. If the app supports a deterministic new-game flow, encode that flow in Playwright.
5. If the app requires fixture data, URL params, debug controls, or seed selection to reach this state, document the exact mechanism in the QA docs and use it in Playwright.
6. Do not leave “load an appropriate game state” implicit. The final QA docs must spell out the exact clicks, form inputs, route, seed, or fixture the reviewer uses.

### Playwright end-to-end validation
1. Create or update a Playwright QA procedure that launches the app and navigates to the exact target game state.
2. Record video for the full session.
3. Verify the initial loaded screen:
   - all four seats render hand rail containers
   - the local seat shows visible hand cards in a rail
   - hidden opponent seats show card backs in rail slots, not a fan arc
   - opponent commanders appear in the same rail as concealed hand slots
   - no detached featured commander card appears
4. Verify interaction behavior in-product:
   - hover a visible card in the local rail and confirm it lifts/scales without clipping
   - move across adjacent visible cards and confirm hover focus follows the active card
   - hover a hidden opponent placeholder and confirm it does not lift, preview, animate, or change pointer behavior
   - attempt clicking hidden placeholders and confirm no preview, dialog, or action is triggered
   - if drag behavior exists for visible hand cards, confirm hidden placeholders do not start drag behavior
5. Verify overflow behavior:
   - load or create a state with enough cards to overflow at least one rail
   - scroll the rail horizontally
   - confirm card backs and visible cards remain aligned and readable
6. Verify off-hand zone integration if a suitable state exists:
   - confirm playable `EXILE` and `GRAVEYARD` cards appear in the rail after hand items
   - confirm hidden placeholders are not treated as zone-anchor cards
7. Save the Playwright video artifact and reference it in the QA doc or review notes.

### Explicit on-screen expectations for Playwright review
- Opponent hand areas should look like straight horizontal rails, not curved fans.
- Card backs should occupy the same slot geometry as visible hand cards.
- A visible commander should appear inline with the concealed hand slots rather than separated or featured.
- Hovering a visible card should create motion only on that visible card.
- Hovering or clicking a hidden card back should leave the screen visually unchanged except for any normal cursor movement.
- When the rail overflows, the reviewer should see horizontal scrolling rather than compressed cards or clipped overflow.
